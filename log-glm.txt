PHASE 1 — Menu CRUD + Publish/Data Quality Gate (P0 blocker pilot)
1A. Migration 0006_admin_menu_write_policies.sql (security-critical)
Saat ini TIDAK ADA INSERT/UPDATE/DELETE policy untuk tabel menu_* → RLS deny-all. Tambah:

menu_categories: INSERT/UPDATE/DELETE USING (app.has_restaurant_access(restaurant_id)) + WITH CHECK (...).
menu_items: sama, scoped restaurant_id.
menu_item_translations, menu_item_dietary_flags, menu_item_allergens: scoped via EXISTS subquery ke menu_items.restaurant_id (join table tak punya kolom restaurant). Sama persis pola RLS SELECT yang sudah ada di 0001.
Semua DROP POLICY IF EXISTS dulu (idempotent), CREATE POLICY. Grant tetap lingua_app.
Catatan: menu_categories UNIQUE (menu_id, name) — tidak berubah.
1B. Migration 0007_menu_item_audit_columns.sql
Tambah updated_at timestamptz + trigger app.set_updated_at() di menu_items dan menu_categories (saat ini tidak punya — pakai pola trigger yang sudah ada di 0001 line 327-385). Audit + bantu debug apakah cache publik perlu di-invalidate. Idempotent (ADD COLUMN IF NOT EXISTS, DROP TRIGGER IF EXISTS).

Export types UpdateMenuItemInput, PublishMenuInput.

name non-empty, price >= 0.

publishFallbackEvent(payload) — getRedisClient().publish('fallback:events', JSON.stringify(payload)) di channel global lalu client subscribe filter by restaurant. Fail-open: subscribe error → log, tidak crash page. Publikasi error → log (guest write tetap sukses, staff manual refresh).

listFallbackRequests(client, { organizationId, restaurantIds, statusFilter? }) — SELECT fallback_requests JOIN restaurants (slug/name) JOIN restaurant_tables (code) WHERE restaurant_id = ANY($restaurantIds) ORDER BY created_at DESC LIMIT 100. Pakai withUserContext.
updateFallbackStatus(client, { organizationId, restaurantId, fallbackId, status, membershipId }) — UPDATE ... WHERE id + restaurant_id + org_id. Set resolved_at = now() kalau status='resolved', assigned_membership_id kalau in-progress. Return updated row.

Update StaffRequest type: tambah field DB-level (createdAt: string, restaurantId, restaurantName, tableId) sambil keep field display.

transitionStatus(user, { fallbackId, status, restaurantSlug }) — resolve tenant → validasi fallback termasuk scope → updateFallbackStatus di withUserContext. Setelah sukses, publish Redis event { type: 'status-changed', fallbackId, status, restaurantId, orgId }.
